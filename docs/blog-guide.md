# Genesis 技术博客撰写指南

本文档是 Genesis 系列技术博客的默认写作规范；如有明确理由，可在不破坏整体风格的前提下微调。

适用范围：组件 blog（如 `genesis-clog-blog.md`）应严格遵循本规范；专题实践文（如可观测性、架构总览）可参考整体调性和行文规范，但不强制套用大纲模板；纯规范文档和设计决策记录不适用本文档。

---

## 1 写作定位

Genesis blog 面向两类读者。第一类是作者本人，写作过程本身就是对基础知识、API 设计和工程取舍的沉淀；第二类是外部开发者，他们需要判断这个组件是否适合自己的项目，并快速掌握推荐用法。因此每篇文章必须同时满足"能学到东西"和"能辅助决策"两个目标。

Genesis 的文档体系分为三个角色，各有分工。README 是入口页，解决"这是什么、怎么快速上手"；Blog 是设计文档，解决"为什么这样设计、适合什么场景、如何做技术决策"；`go doc` 和代码注释是接口文档，解决"具体 API 怎么调用"。三者不应互相替代——README 不负责讲深层设计权衡，Blog 不负责穷举 API，`go doc` 不负责讲场景和架构取舍。

---

## 2 标题与开头

标题统一使用 `Genesis <主题>：<一句话定位>` 格式，其中主题可以是组件名（如 clog、cache），也可以是跨组件专题（如可观测性、设计原则）。标题必须包含 Genesis，副标题体现设计主题或核心定位。推荐风格如"Genesis clog：基于 Slog 的结构化日志组件设计与实现""Genesis cache：统一缓存抽象的设计与实现""Genesis 可观测性：Metrics、Logging 与 Tracing 的集成实践"。不必每篇都以"的设计与实现"收尾，但也不要偏离太远——避免写成"使用说明""源码分析""功能介绍"这类过泛的表达。

文章开头用一段自然文字讲清三件事：它是什么（在 Genesis 架构中的层级和核心职责）、它解决什么问题（面向什么场景）、这篇文章会讲什么（设计取舍，而不只是用法）。对于复杂组件或长文，建议在开头后补一个摘要段落，用 3 到 6 条短句提炼全文核心观点，帮助读者快速建立预期；短文或问题聚焦的文章可以省略摘要，避免与开头重复。

---

## 3 统一大纲

组件文章原则上遵循以下结构，不同组件可以微调章节粒度，但不要大幅偏离整体骨架。

**背景与问题**是文章的起点。先讲清楚这个组件要解决的真实工程问题是什么，直接使用底层库会遇到什么痛点，为什么需要 Genesis 再抽象一层。好的背景段落能让读者在看到任何代码之前就理解这个组件存在的意义。

**基础原理与前置知识**（可选）。当组件强依赖某个底层原理时——例如 auth 依赖 JWT 规范、ratelimit 依赖令牌桶算法、idgen 依赖 Snowflake 位布局、registry 依赖服务发现的 watch 机制——建议单独用一节把这些前置知识讲清楚。这一节的目的是为后续的设计决策和实现思路铺路，避免读者在"背景"和"实现"之间出现认知跳跃。如果组件的工程性强于原理性（如 connector、cache），可以省略这一节，将必要的原理知识融入后续章节。

**设计目标**为后文所有决策提供判断标准。一般 3 到 5 条，例如显式依赖注入、统一接口、可观测性友好、低心智负担。这一段不需要长篇论证，简洁列出即可，但每条都应该在后文的设计与取舍中有所呼应。

**核心接口与配置**帮助读者建立"怎么用"和"为什么这么用"的第一层理解。展示最关键的公开接口、构造方式、Config 结构和 Option 模式，重点解释为什么接口这样设计而不是做得更大更复杂。配置项适合用表格呈现，接口定义适合用代码块。

**核心概念与数据模型**帮助读者建立稳定的心智模型。包括组件最核心的概念、状态流转、资源所有权、错误模型、关键对象之间的关系。不是所有组件都需要单独成节，如果概念简单可以合并到上一节。

**关键实现思路**解释架构而不是逐文件游览源码。聚焦实现主链路和与设计目标直接相关的内部结构，讲清楚"怎么做到的"，而不是"代码长什么样"。涉及的底层原理（算法、协议、标准）如果没有在前置知识一节中展开，可以在这里穿插说明。

**工程取舍与设计权衡**是 Genesis blog 区别于普通技术文档的核心段落。为什么选这个方案而不是另一个，当前故意不做什么，简单性、扩展性、性能、稳定性之间如何平衡。这一段展示的是技术判断力，而不是功能清单。

**适用场景与实践建议**帮助读者做最终的技术决策。包括适合什么项目和负载、不适合什么场景、什么时候应该直接用底层库而不是 Genesis 组件、生产环境和开发环境的推荐配置、常见误区和正确做法。

**总结**收束全文，用几句话提炼组件的核心价值、最重要的使用原则以及它在 Genesis 中的长期意义。如果有明确的演进方向可以简要提及，但不要把未实现的能力写成既有能力。

---

## 4 内容侧重

每篇文章的重点顺序应当是：问题背景、设计目标、接口设计、工程取舍、推荐实践、实现细节。这意味着实现细节很重要但不能喧宾夺主，设计与取舍必须先于源码分析，不要把文章写成"我读了哪些文件"的过程记录。

作为经验参考，多数组件文章的内容分布大致为：背景、定位和目标占 30%，API、概念和设计取舍占 30%，实现思路占 20%，场景、误区和实践占 20%。但这不是硬指标——不同组件天然重心不同，auth 和 ratelimit 的基础原理部分会更重，clog 和 connector 的工程取舍部分会更厚，应根据组件特点灵活调整。

如果一篇文章没有回答清楚以下五个问题，说明结构还不够成熟：它解决什么问题？为什么不直接用底层库？为什么 API 这样设计？它适合什么场景、不适合什么场景？使用者最容易踩什么坑？

## 5 行文规范

### 语气与措辞

整体语气应当专业、克制、直接。少用宣传腔，多写"为什么"。禁止使用表情符号和感叹号表达情绪。避免无依据的价值判断——如果使用"简洁""高性能""优雅"等评价性词汇，必须紧跟可验证的理由（例如"接口更简洁，因为将五个方法收敛为两个"就是合理的）；没有支撑的空泛修饰如"完美解决""业界领先"则应删去。同一概念在全系列中必须使用同一术语，避免在同一篇文章里频繁切换同义词。

### 表达方式

对于流程、原理、机制等说明性内容，优先使用自然段落表述，让逻辑连贯地流动，而不是拆成碎片化的短列表。列表仅在以下场景使用：配置项说明（配合表格）、并列的短项（如 3 到 6 个设计目标）、步骤性操作。代码块仅在以下场景使用：完整的代码示例、API 接口定义、配置文件示例、数据结构定义。避免用代码块展示可以用自然语言描述的运行逻辑。

### 结构顺序

行文遵循四个先后原则：先问题后方案，先边界后能力，先设计后实现，先取舍后细节。不要上来就从源码开始写，否则文章会自然滑向"实现笔记"而不是"设计文章"。

### 示例代码

示例必须短小精悍，代表推荐用法，并与当前真实实现一致。不要把所有 API 都堆进一个示例，每个示例聚焦一个场景。

---

## 6 格式规范

一级标题使用 `# Genesis <主题>：<设计主题>`，二级标题使用 `## <编号> <主题>`（如 `## 1 背景`），三级标题使用 `### <子主题>`（不编号）。段落间空一行，段落首行不缩进，段落内不使用强制换行。关键概念使用**粗体**强调，术语首次出现可用粗体，代码、命令和文件名使用行内代码块。引用规范或标准时使用 `[RFC 7519]` 格式，引用其他文档使用相对链接。主要章节间可用 `---` 分隔，但不要过度使用。

---

## 7 应避免的写法

不要把 blog 写成 README 的扩写版、`go doc` 的翻版或源码逐段解释。不要在文章开头就进入实现细节。不要花大量篇幅描述"未来可能扩展"的能力——如果当前没有实现，点到为止即可。不要宣称"零成本""零开销""通用万能"，除非实现确实支撑得起这些断言。不要把文章写成纯 API 罗列或纯宣传文案。

---

## 8 写作流程与检查清单

建议按以下顺序写作：先写组件定位和问题背景，再写设计目标和接口设计，然后写关键取舍和实现思路，最后补推荐用法、误区和总结。

发布前检查以下内容：标题是否符合统一格式；开头是否讲清定位、问题和文章目标；是否明确写出组件边界（能做什么、不能做什么）；是否解释了设计目标和关键取舍；示例代码是否与当前实现一致；文中 API、字段、配置和行为是否与真实代码一致；是否包含适用场景与不适用场景；是否包含常见误区；原理和机制说明是否优先使用了自然段而非碎片列表；无表情符号、无感叹号；评价性词汇是否有理由支撑；术语使用是否与 glossary.md 一致；章节编号连续。
