# Genesis xerrors：轻量错误封装组件的设计与取舍

Genesis `xerrors` 是一个 Level 0 基础组件，核心职责不是重新发明 Go 的错误处理模型，而是在保持标准库 `errors` 语义不变的前提下，补齐几个高频、稳定、工程上值得统一的小能力。这篇文章重点讨论它为什么被设计成一个很薄的组件、它适合什么场景，以及它刻意不做什么。

## 0 摘要

`xerrors` 的设计核心不是“更强”，而是“更稳”。它围绕标准库错误链，只提供上下文包装、轻量错误码、初始化断言和简单的多错误处理。它不提供 stack trace、不统一建模所有协议错误、也不试图成为一套完整的错误框架。这样的克制让它更适合作为 Genesis 的 L0 基础组件。

---

## 1 背景与问题

Go 标准库的 `errors` 已经足够好。`errors.Is`、`errors.As`、`errors.Unwrap` 和 `errors.Join` 为错误链提供了清晰、一致且语言层面支持的语义。在这样的前提下，再造一套错误系统通常得不偿失。

但在工程实践中，仍然存在几个反复出现的小问题。第一，业务代码需要频繁在底层错误上追加上下文，如果每次都手写 `fmt.Errorf("%s: %w", ...)`，样板代码会很多。第二，部分业务场景确实需要一个轻量的机器可读错误码，用于上层协议映射或错误分类。第三，初始化阶段经常会出现“失败即 panic”的依赖装配逻辑。第四，顺序校验流程里经常只想保留第一个错误，或者把多个错误合并成一个返回值。

`xerrors` 就是在这个背景下存在的。它不是因为标准库不够用，而是因为 Genesis 希望在不破坏标准库心智模型的前提下，把这些高频小能力收敛到一个统一入口。

## 2 设计目标

`xerrors` 的设计目标很少，而且边界明确。

- 保持与标准库 `errors` 的行为一致，不制造第二套错误语义。
- 用极小的 API 面补齐高频工程能力，而不是构建完整错误框架。
- 让错误链、哨兵错误和类型匹配依然是第一原则。
- 为 Genesis 其他组件提供统一的轻量错误工具，但不反向侵入业务协议层。

这几个目标直接决定了 `xerrors` 最终会很薄，也决定了它不会提供很多看起来“很强”的能力。

## 3 核心接口与能力边界

`xerrors` 的公开接口非常小，真正新增的能力只有几类。

```go
func Wrap(err error, msg string) error
func Wrapf(err error, format string, args ...any) error

func WithCode(err error, code string) error
func GetCode(err error) string

func Must[T any](v T, err error) T
func MustOK[T any](v T, ok bool) T

type Collector struct
func (c *Collector) Collect(err error)
func (c *Collector) Err() error

func Combine(errs ...error) error
```

除此之外，`xerrors` 只对标准库 `errors` 做了再导出，方便 Genesis 组件统一从一个入口使用 `New`、`Is`、`As`、`Join` 和 `Unwrap`。

这里最重要的是边界。`xerrors` 不负责：

- stack trace 采集
- 统一错误分类体系
- 协议层错误建模
- 并发安全的错误聚合
- 自动把错误映射到 HTTP 或 gRPC 响应

如果把这些需求都压进一个 L0 组件，它很快就会从“基础工具”变成“框架式错误系统”，这和 Genesis 整体强调的显式、克制、可组合相冲突。

## 4 核心概念与数据模型

`xerrors` 的核心概念只有三个：错误链、轻量错误码和简单聚合。

第一是**错误链**。`Wrap` 和 `Wrapf` 的本质只是给错误追加上下文，并保留 `%w` 语义。这意味着上层依然可以继续使用 `errors.Is` 判断哨兵错误，也可以使用 `errors.As` 提取具体类型。`xerrors` 并不改变这条链，只是在链上附加更多信息。

第二是**轻量错误码**。`WithCode` 会把一个字符串 `code` 附加到错误链上，`GetCode` 则通过 `errors.As` 取出链中的第一个 `CodedError`。这里的设计非常克制：它只表达一个机器可读错误码，不负责 `retryable`、`http_status`、`grpc_code`、`public_message` 等更丰富的元数据。这样做的好处是简单，但代价也很明确：如果系统需要统一的协议错误建模，应该在更高层单独设计，而不是把所有需求都压给 `xerrors`。

第三是**简单聚合**。`Collector` 的语义不是“收集所有错误”，而是“保留第一个非 `nil` 错误”。它适合顺序校验流程，例如表单校验、配置校验或构造参数校验。`Combine` 则是一个更中性的工具，把多个错误折叠成一个返回值；当错误超过一个时，它返回 `MultiError`，并通过 `Unwrap() []error` 兼容标准库的多错误匹配语义。

## 5 关键实现思路

从实现上看，`xerrors` 几乎没有复杂结构。`Wrap` 和 `Wrapf` 只是对 `fmt.Errorf(... %w ...)` 的轻量封装；`CodedError` 实现了 `Error()` 和 `Unwrap()`，因此天然兼容标准库错误链；`GetCode` 直接通过 `errors.As` 提取 `*CodedError`。这意味着它没有引入新的匹配规则，也不要求调用方学习新的错误遍历方式。

`Collector` 的实现也刻意保持简单：内部只保留一个 `err` 字段，第一次看到非 `nil` 错误就记录下来，后续错误全部忽略。这个选择并不复杂，但很重要，因为它把行为定义得足够窄，调用方不会误以为它会保留全部错误。`Combine` 同样遵循最小规则集：过滤 `nil`，零个返回 `nil`，一个返回原错误，多个返回 `MultiError`。这种实现方式和 Go 标准库的思路是一致的，不追求“魔法”，只追求可预测。

## 6 工程取舍与设计权衡

`xerrors` 最大的设计取舍，是**刻意不扩展成完整错误框架**。这背后有几个原因。

首先，Go 社区在错误处理上已经有稳定共识：错误链、哨兵错误、具体类型匹配和显式返回值。只要标准库已经能稳定解决的问题，就没有必要再包一层复杂抽象。`xerrors` 如果继续扩展 stack、分类、协议映射、公共元数据，很容易变成另一套“必须学习的新系统”，这会抬高整个仓库的使用门槛。

其次，错误模型天然和业务边界强相关。缓存、认证、消息队列、数据库代理，这些组件对错误的分类需求并不相同。把所有分类需求提前抽象到 L0 层，通常只会得到一个既不够强又不够清晰的通用模型。Genesis 在这里选择把 L0 层做薄，把更具体的错误策略留给上层组件。

最后，`Collector` 和 `Combine` 也体现了同样的取舍。它们只解决最常见的两个需求：顺序保留第一个错误，和把多个错误折叠成一个返回值。更复杂的聚合需求，例如保留所有错误并带字段路径、并发安全聚合、按严重级别分类，都应该在上层单独建模。

## 7 适用场景与实践建议

`xerrors` 最适合这些场景：业务层和组件层需要保留标准库错误链，同时统一错误包装写法；上层协议需要从错误中提取一个轻量错误码；初始化代码希望用 `Must` 明确表达“失败即 panic”；顺序校验流程只关心第一个错误或需要把多个错误合并成一个返回值。

它不适合这些场景：你需要 stack trace 自动采集；你需要统一的公共错误结构并跨协议传播；你需要并发安全的错误汇聚；你希望通过一个基础组件直接完成 API 错误响应建模。

实践上有几条建议值得明确。第一，哨兵错误仍然应该用 `xerrors.New` 定义，再通过 `errors.Is` 或 `xerrors.Is` 判断，而不是用字符串比较错误文本。第二，`WithCode` 应该只在确实需要机器可读码时使用，不要把普通业务错误全部编码化。第三，`Collector` 只适合顺序流程，若需要保留全部错误，应直接设计更适合业务的数据结构。第四，`Must` 和 `MustOK` 只用于初始化或测试，运行时路径继续坚持显式返回错误。

## 8 总结

`xerrors` 的价值不在于功能很多，而在于它为 Genesis 提供了一套**与标准库一致、接口极小、行为稳定**的错误辅助工具。它解决的是工程里的高频小问题，而不是试图成为整个系统的错误治理中心。对一个 L0 组件来说，这种克制比“看起来能力更强”更重要。
