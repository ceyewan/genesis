# Genesis idem：结果复用型幂等组件的设计取舍

Genesis `idem` 是业务层组件，用来处理“同一请求被重复提交”或“同一消息被重复投递”这类问题。它的核心思路并不神秘：先判断是否已有成功结果，如果有就直接复用；如果没有，就用锁保证同一 key 只有一个执行者，然后在成功后写入结果缓存。这篇文章重点讲的是它为什么被定义成“结果复用型幂等组件”，以及这一定义带来的边界与取舍。

---

## 0 摘要

- `idem` 的核心语义是防重复成功提交，而不是严格的 exactly-once 执行。
- 组件通过“查结果、抢锁、执行、写结果”的主链路统一服务于 `Execute`、`Consume`、Gin 中间件和 gRPC 拦截器。
- `Execute` 现在会把首次成功执行和缓存命中统一成同一套 JSON 规范化结果，避免返回类型漂移。
- HTTP 和 gRPC 的缓存策略被显式化了：HTTP 按状态码决定是否缓存，gRPC 按 `proto.Message` 及可选策略决定是否缓存。
- 锁续期失败不再只是 warning，而会被视为真实风险并阻止结果继续写入缓存。

---

## 1 背景与问题

幂等常常被当作一句业务约定来讲，但真正落到系统里，它处理的是一个很具体的问题：调用方重试了，或者消息中间件重投了，而你不希望成功结果被提交两次。典型例子包括重复创建订单、重复扣款、重复发券、重复消费消息。

最原始的做法通常是业务自己写一套去重：查数据库、写去重表、用 Redis 做 `SETNX`、或者在每个 handler 里手动判断“这个 request_id 有没有处理过”。这些做法不是不能用，但会带来三个问题。第一，幂等逻辑会散落在各个业务入口。第二，很多实现只做了“抢锁”，没有把成功结果缓存下来，导致并发请求仍然需要自己处理等待和结果复用。第三，HTTP、gRPC、消息消费这三类入口经常各写一套，行为很难保持一致。

Genesis `idem` 想解决的正是这三个问题：把主链路标准化，把结果复用纳入组件语义，并把多入口场景接到同一套核心逻辑上。

---

## 2 基础原理与前置知识

`idem` 的主链路可以概括为四步：查结果、抢锁、执行、写结果。一个 key 有三种状态。第一种是“没有结果，也没有执行者”，此时当前请求拿到锁并执行业务逻辑。第二种是“已经有成功结果”，此时当前请求直接复用缓存结果。第三种是“正在执行但尚未写出结果”，此时当前请求不能再并发执行，只能等待结果、继续轮询，或者在某些场景下直接返回并发错误。

这套模型天然适合“成功结果可复用”的场景，也天然不适合“每次执行都有副作用但又没法缓存结果”的场景。这也是为什么 `idem` 的边界要从一开始就说清楚：它不是事务协调器，不是数据库唯一约束的替代品，也不是严格意义上的 exactly-once 执行协议。

---

## 3 设计目标

`idem` 当前的设计目标有四个。第一，用一套统一主链路覆盖手动调用、HTTP 和 gRPC 三类主要入口。第二，把成功结果复用纳入幂等语义，而不是只做“执行中加锁”。第三，支持 Redis 和 Memory 两种后端，让组件既能用于分布式服务，也能用于本地开发和测试。第四，在不引入过重抽象的前提下，把最危险的边界显式化，例如缓存策略和锁续期失败。

这几个目标共同决定了它今天的样子：比简单 `SETNX` 方案更完整，但仍然是应用层幂等组件，而不是底层事务系统。

---

## 4 核心接口与配置

`idem` 的对外接口有四个入口。`Execute` 面向通用业务逻辑，返回业务结果；`Consume` 面向消息消费，只告诉调用方“这次有没有真正执行”；`GinMiddleware` 和 `UnaryServerInterceptor` 则把同一套逻辑接到 HTTP 和 gRPC 服务端入口。

配置项本身不复杂。`DefaultTTL` 决定成功结果会被缓存多久，`LockTTL` 决定执行阶段的锁有效期，`WaitTimeout` 和 `WaitInterval` 决定等待结果时的行为。这里一个看起来不起眼但很重要的调整是：负数配置现在会被显式拒绝，而不是静默回退默认值。对幂等组件来说，最怕的不是参数保守，而是调用方以为自己设置了特殊语义，结果组件悄悄替他改回默认值。

---

## 5 核心概念与数据模型

这个组件最关键的概念是“成功结果缓存”。也就是说，`idem` 真正复用的是“已经成功提交过的结果”，而不是“执行过就算完成”的痕迹。对 `Execute` 来说，这意味着成功结果会被序列化后缓存下来；对 `Consume` 来说，则意味着成功后会写一个“已处理”标记，而不是保留业务返回值。

这也是为什么我们把组件定位从“exactly-once”收紧到了“防重复成功提交”。失败结果不会缓存，后续允许重试；锁续期失败时，也不能继续假装当前执行还拥有完整的排他性保证。

另一个关键概念是“返回值规范化”。过去 `Execute` 的第一次成功执行会直接返回原始 Go 值，而缓存命中则返回 JSON 反序列化后的通用结构，导致第一次是结构体、第二次变成 `map[string]any`。现在这两条路径都统一走一套 JSON 编解码，这让返回值虽然仍然是 `any`，但至少不再在“首次执行”和“缓存命中”之间漂移。

---

## 6 关键实现思路

核心实现仍然围绕那条主链路展开。`Execute` 先尝试获取已有结果，未命中时再尝试拿锁。拿到锁之后执行业务函数，成功后把结果做 JSON 规范化并缓存，再返回规范化后的结果；失败则直接释放锁，不写缓存。由于第一次返回和缓存命中都使用同一套解码过程，返回值类型终于稳定了下来。

这次还补了一个很现实的恢复路径：如果读取到的缓存内容已经损坏，组件会尝试删除坏缓存并重新进入执行流程，而不是直接长期卡在“这个 key 以后永远只能报错”的状态。这个恢复逻辑本身并不复杂，但很重要，因为缓存损坏在 Redis、手工写入或版本演进场景下都不算纯理论问题。

HTTP 和 gRPC 的接入层也做了收敛。Gin 中间件现在不再把“只缓存 2xx”隐藏成写死行为，而是通过 `WithHTTPStatusCacheFunc` 变成显式策略。默认仍然是缓存 `2xx`，但如果某些 `4xx` 在你的系统里属于稳定、可复用的业务结果，就可以明确配置进去。gRPC 这边的边界更明确：当前依然只缓存成功的 `proto.Message` 响应，但你可以通过 `WithGRPCResponseCacheFunc` 缩小缓存范围，而不是只能接受一个写死的策略。

最后一个关键实现点是锁续期。过去续期失败只会记 warning，主流程继续跑，结果是本地还在执行，但锁可能已经在远端失效。现在续期失败会被视为真实错误，至少在 `Execute` 和 `Consume` 这类直接调用路径上，它会阻止成功结果继续写入缓存。对 HTTP/gRPC 这类 handler 已经可能开始写响应的场景，这个保证没法做得一样强，但至少组件不会再默默把一个已经失去排他性的结果缓存下来。

---

## 7 工程取舍与设计权衡

这里最值得讲的取舍有两个。第一个是为什么没有直接把 `Execute` 做成泛型强类型接口。答案是：从 API 体验上看那确实更理想，但它会把当前组件从“通用幂等层”推向“带编解码协议的类型化缓存层”，这一步会明显扩大公开接口改动面，也会让 Gin/gRPC 接入层一起重做。当前这轮选择的是更保守的路线：先把返回值稳定下来，再把强类型恢复留给后续可能的 codec 设计。

第二个是为什么 gRPC 仍然只缓存 `proto.Message`。原因很简单：服务端拦截器可以比较可靠地缓存和恢复 protobuf 响应，但无法通用地恢复任意 Go 值。这里如果继续假装“所有成功结果都能缓存”，只会让组件的语义越来越虚。与其给一个看上去更通用、实际处处例外的接口，不如先把边界说清楚。

---

## 8 适用场景与实践建议

如果你的目标是“同一个 request_id 的成功结果只能提交一次”，而且返回值本身可以被缓存复用，那么 `idem` 很合适。HTTP 下单、gRPC 创建资源、消息消费去重，这些都是典型用法。Redis 适合分布式部署，Memory 适合单机和测试。

如果你的目标是“强类型地恢复任意复杂 Go 结果”，那么当前 `Execute` 还不够理想，因为它虽然已经稳定了结果形态，但仍然是以 JSON 友好的通用值为主。对于这类场景，更好的方向通常是业务显式定义自己的编解码器，而不是让组件猜测你的目标类型。

另一个实践建议是，不要把缓存策略当成无关紧要的细节。某些系统里，`409 Conflict` 或某类业务性 `4xx` 本身就是可以安全复用的结果；另一些系统里，它们又不应该缓存。同样的道理也适用于 gRPC。幂等组件最危险的不是没提供默认值，而是把策略写死后让调用方根本无从表达自己的业务判断。

---

## 9 总结

Genesis `idem` 的核心价值，不在于它宣称自己能做严格的 exactly-once，而在于它把应用层幂等真正需要的三件事组合到了一起：成功结果复用、并发锁保护，以及多入口统一接入。

这轮收敛之后，它的定位更清楚了：`idem` 是结果复用型幂等组件，不是事务系统；它能显著降低重复成功提交的风险，但不会替应用承担一切分布式一致性问题。真正用好它的关键，在于设计好 key、理解好缓存边界，并接受它是应用层幂等，而不是魔法。*** End Patch
