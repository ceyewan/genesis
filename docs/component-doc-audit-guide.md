# Genesis 组件文档审计规范

本文档定义 Genesis 组件文档审计时的统一标准，适用于组件包注释与 `go doc`、组件目录下的 `README.md`，以及 `docs/` 下的组件 blog。

本文档的目标不是替代 [总体设计](./genesis-design.md) 或 [blog 撰写指南](./blog-guide.md)，而是把本轮“全组件文档审计”需要执行的检查项、职责边界和例外情况收敛成一份可复用的操作规范。

---

## 1 适用范围

默认适用对象为以下正式组件：

- `auth`
- `breaker`
- `cache`
- `clog`
- `config`
- `connector`
- `db`
- `dlock`
- `idgen`
- `idem`
- `metrics`
- `mq`
- `ratelimit`
- `registry`
- `trace`
- `xerrors`

`testkit` 属于测试辅助包，不是正式生产组件。它需要遵守 `go doc` 与 README 规范，但默认不要求补齐组件 blog；除非后续明确决定将其提升为需要长期维护设计文章的正式组件。

---

## 2 三层文档职责

Genesis 组件文档分为三层，职责不能混淆。

### 2.1 `go doc` / 包注释

`go doc` 负责定义组件定位、主接口、行为语义和能力边界。它必须回答“这个包能做什么、不能做什么、核心对象如何使用”。

`go doc` 必须低于或等于实现，不能写出 README 或 blog 才出现的更强承诺，也不能把未来计划写成当前能力。

### 2.2 README

README 是组件入口页，负责快速上手、推荐接入方式、常见配置和实践建议。它必须回答“我该怎么接入、应该怎么用、最容易踩什么坑”。

README 不负责展开深层设计论证，不应写成 blog 的缩写版，也不应变成完整 API 索引。

### 2.3 Blog

组件 blog 是设计文档，负责解释为什么这样设计、采用了哪些工程取舍、适合什么场景、不适合什么场景，以及当前刻意不做什么。

blog 不应复述全部 API，也不应逐文件讲源码，更不能使用宣传性表述替代技术判断。

---

## 3 通用审计原则

所有组件的三层文档都必须遵守以下原则：

1. 文档语义不得高于实现。
2. 术语必须一致，同一概念不得在不同文档中随意换名。
3. 资源所有权必须写清楚，尤其是 `Close()`、watcher、session、subscription、lease、watchdog 这类长期资源。
4. 多后端组件必须写清楚驱动差异，不能伪装成完全等价的统一语义。
5. 错误边界必须稳定，组件自定义错误优先于底层库错误直出。
6. 推荐实践必须和当前实现一致，示例代码必须能反映真实构造方式和资源释放方式。
7. 未实现能力、未来计划和潜在扩展不得写成现有能力。
8. 示例优先短小、单场景、可复制，不堆叠全部 API。

---

## 4 `go doc` 审计要求

每个正式组件都必须具备可通过 `go doc -all ./<component>` 导出的稳定包注释。包注释至少应包含以下内容：

1. 组件定位。
2. 所属层级和核心职责。
3. 主接口或主要入口。
4. 能力边界。
5. 资源语义。
6. 多后端差异或全局副作用。
7. 一段最小可行示例。

`go doc` 重点检查项如下：

- 是否明确说明组件属于 L0/L1/L2/L3 的哪一层。
- 是否明确说明统一入口下的真实语义，而不是只写理想抽象。
- 是否说明 `Close()` 是 no-op、必须调用，还是只负责组件自建资源。
- 是否说明全局模式、副作用安装、单实例约束或进程级限制。
- 是否说明 `ErrMiss`、`ErrNotSupported`、`ErrOwnershipLost` 这类关键错误的触发语义。
- 是否避免把 README 的场景介绍或 blog 的大段设计取舍塞进包注释。

如果组件存在明显子能力边界，也应在包注释中直接说清。例如：

- `cache` 要明确 `Distributed`、`Local`、`Multi` 的能力差异。
- `dlock`、`ratelimit`、`mq`、`registry` 要明确不同后端或模式的差异。
- `metrics`、`trace` 要明确“全局模式”及其覆盖行为。
- `db` 要明确借用 connector 生命周期。

---

## 5 README 审计要求

每个正式组件和 `testkit` 都必须有 README。README 的推荐结构允许微调，但通常应包含以下部分：

1. 组件定位。
2. 快速开始。
3. 核心接口或核心能力。
4. 配置说明或关键选项。
5. 推荐实践。
6. 资源释放、错误语义或使用边界。
7. 相关文档链接。

README 重点检查项如下：

- 标题是否清晰标识组件名。
- 开头是否用 1 到 2 段说明组件是什么、适合什么场景、指向 blog 与 `go doc`。
- 示例是否体现推荐用法，而不是仅展示“能跑”的最低代码。
- 是否明确“不适合什么场景”或“当前不支持什么”。
- 是否明确依赖哪些 connector 或外部前置条件。
- 是否明确资源释放责任和组件关闭顺序。
- 是否提供指向对应 blog 和 `go doc` 的入口。

README 中应优先强调“如何正确使用”，不应承担完整设计论证。若出现大量“为什么不这么做”的论述，应转移到 blog。

---

## 6 Blog 审计要求

正式组件原则上都应有对应 blog，并遵循 [blog 撰写指南](./blog-guide.md)。组件 blog 至少应覆盖以下主题：

1. 背景与问题。
2. 基础原理与前置知识（如适用）。
3. 设计目标。
4. 核心接口与配置。
5. 核心概念与数据模型。
6. 关键实现思路。
7. 工程取舍与设计权衡。
8. 适用场景与实践建议。
9. 总结。

blog 重点检查项如下：

- 标题是否符合 `# Genesis <主题>：<一句话定位>`。
- 开头是否同时讲清定位、问题和文章目标。
- 是否明确写出“当前故意不做什么”。
- 是否解释为什么不直接暴露底层库，或为什么不继续做更强抽象。
- 是否把关键设计判断和实现细节分开叙述。
- 是否说明适用场景和不适用场景。
- 是否避免把未来规划写成既有能力。

`docs/genesis-observability-blog.md` 属于跨组件专题文，可参考整体调性，但不强制按组件 blog 模板审计。

---

## 7 三层一致性检查

审计时不能只分别润色三层文档，必须做交叉一致性检查。每个组件至少检查以下问题：

1. `go doc` 的定位是否与 README 第一段一致。
2. README 的快速开始是否与当前公开 API 一致。
3. blog 中的设计目标、边界和取舍是否与 `go doc` 的能力边界一致。
4. 多后端差异是否同时出现在 `go doc`、README 和 blog 中，只是详略不同。
5. README 或 blog 是否承诺了代码中不存在的行为。
6. README 和 blog 中的错误语义是否与包内 sentinel error 一致。
7. 资源释放顺序和关闭责任是否在三层文档中表达一致。

如果三层文档发生冲突，优先级应为：实现 > `go doc` > README > blog。修订方向应始终是让文档向实现收敛，而不是让措辞向更强能力漂移。

## 8 组件类型专项检查

不同类型组件在审计时还需补充以下专项检查。

### 8.1 L0 基础组件

适用于 `clog`、`config`、`metrics`、`trace`、`xerrors`。

重点检查：

- 是否明确“统一约束”而非“隐藏全部底层能力”。
- 是否说明全局安装、副作用、默认值和关闭行为。
- 是否避免把工具包写成框架。

### 8.2 L1 基础设施组件

适用于 `connector`、`db`。

重点检查：

- 是否明确 connector 是资源所有者，`db` 是借用者。
- 是否明确连接建立、健康检查、关闭与客户端暴露方式。
- 是否避免跨数据库能力的过度统一表述。

### 8.3 L2 业务组件

适用于 `cache`、`idgen`、`dlock`、`idem`、`mq`。

重点检查：

- 是否明确不同驱动或不同模式的真实能力边界。
- 是否说明结果复用、锁所有权、投递保证、ID 语义等关键约束。
- 是否把“通用能力”写成“通用万能能力”。

### 8.4 L3 治理组件

适用于 `auth`、`ratelimit`、`breaker`、`registry`。

重点检查：

- 是否明确请求路径中的治理位置和适用接入层。
- 是否说明失败策略、拒绝策略、恢复语义和单实例约束。
- 是否清楚区分业务错误与治理错误。

---

## 9 并行审计拆分原则

组件审计可以并行，但要满足以下前提：

1. 统一规范已先固定。
2. 每个并行任务有明确组件边界。
3. 若涉及共享索引文档，如 `README.md`、`docs/README.md`，应由主线程统一收口，避免并发冲突。
4. 若一个专题 blog 同时覆盖多个组件，如 observability，应在所有相关组件收敛后统一复核。

推荐按以下分组并行：

- L0 组：`clog`、`config`、`metrics`、`trace`、`xerrors`
- L1 组：`connector`、`db`
- L2-A 组：`cache`、`idgen`
- L2-B 组：`dlock`、`idem`、`mq`
- L3 组：`auth`、`ratelimit`、`breaker`、`registry`
- 测试组：`testkit`

这种拆分方式兼顾了层级相近性、文档风格一致性和文件写入冲突控制。

---

## 10 审计输出要求

每个组件审计完成后，至少应产出以下结果：

1. 是否通过审计。
2. 修订了哪些文件。
3. 发现了哪些文档与实现不一致的问题。
4. 是否还存在待确认风险或需要后续代码配合的问题。

若某组件当前实现本身存在边界不清的问题，审计可以先把文档收紧到“保守且真实”的描述，再单独列出后续代码层面的修正建议。
