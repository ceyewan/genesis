# Genesis dlock：分布式锁组件的设计与取舍

Genesis `dlock` 是 L2 业务层的分布式锁组件，面向“多个实例需要对同一资源做互斥访问”的常见工程场景。它不试图抽象所有锁模型，而是把最稳定、最常用的能力收敛成一个很小的 `Locker` 接口，并分别落到 Redis 和 Etcd 两种后端上。这篇文章重点讲 `dlock` 为什么存在、为什么接口只有这么小、Redis 和 Etcd 的实现分别怎么取舍，以及哪些边界是组件故意不做的。

## 0 摘要

- `dlock` 不是锁平台，而是 Genesis 内部统一分布式锁语义的一层薄抽象
- 对外只暴露 `Lock`、`TryLock`、`Unlock`、`Close` 四个核心方法
- Redis 方案基于 `SET NX`、token 校验和 watchdog 自动续期
- Etcd 方案基于 `concurrency.Mutex` 和 session/lease 模型
- `Close()` 会停止续期并尽力释放当前 `Locker` 已持有的锁
- Redis 和 Etcd 并不是完全等价的协议实现，尤其 TTL 精度边界必须显式理解

---

## 1 背景与问题

在单进程程序里，互斥通常由 `sync.Mutex` 解决；但进入微服务和多实例部署后，资源竞争变成跨进程问题，单机锁就不再够用。定时任务竞选、库存扣减、幂等保护、短事务串行化，这些场景都需要一把“大家都能看见”的锁。

直接使用 Redis 或 Etcd 原生能力当然可行，但调用方很快会遇到一组重复问题：如何防止误删别人的锁，业务执行超过 TTL 怎么续期，拿不到锁时是阻塞重试还是立即返回，进程退出时该怎么清理本地持有状态，如何区分“锁冲突”和“所有权已经丢了”。`dlock` 这一层的价值，不在于发明新算法，而在于把这些高频且容易写错的细节收敛成稳定的组件行为。

---

## 2 基础原理与前置知识

### 2.1 Redis 锁的核心问题

Redis 最常见的分布式锁写法是 `SET key value NX PX ttl`。`NX` 保证 key 不存在时才创建，`PX` 给锁一个自动过期时间，避免持有者崩溃后永远不释放。问题在于，释放锁不能直接 `DEL key`，否则任何客户端都能删掉这把锁。更安全的写法是把一个随机 token 作为 value 写进去，解锁时用 Lua 脚本比较 value，只有 token 匹配才删除。

另一个问题是 TTL。业务执行时间往往不可预测，如果锁的 TTL 太短，业务还没执行完锁就过期了；太长又会放大崩溃后的锁滞留时间。因此 Redis 方案通常还要配一个 watchdog，在锁持有期间自动续期。

### 2.2 Etcd 锁的核心问题

Etcd 的锁能力不是通过自己拼装 KV 命令来实现，而是依赖 lease 和 `concurrency.Mutex`。lease 决定锁的生命周期，只要 session 存活，Etcd 就会自动续期；session 关闭或断开，lease 过期，对应的锁也会释放。

这种模型比 Redis 更接近“租约锁”，但它天然带来一个边界：Etcd lease 的 TTL 是秒级，不支持亚秒精度。因此同样叫 `WithTTL(...)`，Redis 可以接受任意 `time.Duration`，Etcd 则必须至少 1 秒且是整秒。

## 3 设计目标

- 提供一个足够小的统一接口，让大多数业务只需要理解 `Lock / TryLock / Unlock / Close`
- 在 Redis 和 Etcd 上都保证“只有持有者才能释放锁”
- 自动处理续期，减少长任务执行时的锁过期风险
- 把生命周期语义说清楚，`Close()` 真正收尾，而不是空操作
- 对非法 TTL、所有权丢失、本地重入等边界做显式失败，而不是静默回退

这些目标决定了 `dlock` 的设计方向：它优先追求语义可预测，而不是追求 API 花样或极致灵活。

---

## 4 核心接口与配置

对外接口非常克制：

```go
type Locker interface {
    Lock(ctx context.Context, key string, opts ...LockOption) error
    TryLock(ctx context.Context, key string, opts ...LockOption) (bool, error)
    Unlock(ctx context.Context, key string) error
    Close() error
}
```

`Lock` 是阻塞式获取锁，按 `RetryInterval` 重试，直到拿到锁或 `context` 结束。`TryLock` 不阻塞，只告诉你这次有没有抢到。`Unlock` 只做一件事：释放当前 `Locker` 持有的那把锁。`Close` 则是整个组件生命周期的收尾动作，它会停止续期，并尽力释放当前 `Locker` 仍然持有的锁。

配置同样保持最小集合：

| 字段 | 说明 |
| --- | --- |
| `Driver` | 后端类型，当前支持 `redis` 和 `etcd` |
| `Prefix` | 锁 key 前缀，用于命名空间隔离 |
| `DefaultTTL` | 默认锁 TTL |
| `RetryInterval` | `Lock` 重试间隔 |

运行时只暴露一个 `WithTTL(...)` 选项，用于覆盖单次操作的 TTL。这里的设计取舍很明确：不提供复杂的 per-key 策略系统，不把锁组件做成一套 DSL，只保留最常见、最有确定性的一个运行时维度。

## 5 核心概念与错误模型

`dlock` 的核心概念其实只有三个：**远端锁状态**、**本地持有状态**、**续期生命周期**。远端锁状态由 Redis key 或 Etcd mutex/lease 表示；本地持有状态由组件内部 `locks map` 维护，用于阻止同一个 `Locker` 在本地重复持有同一个 key；续期生命周期则由 Redis watchdog 或 Etcd session keepalive 承担。

错误模型围绕这三个概念展开：

- `ErrLockAlreadyHeld`：当前 `Locker` 本地已经持有这个 key
- `ErrLockNotHeld`：调用方试图释放一个当前 `Locker` 没有持有的锁
- `ErrOwnershipLost`：远端锁已经不属于当前持有者
- `ErrInvalidTTL`：TTL 非法，最常见于 Etcd 子秒级或非整秒 TTL

这些错误不是“锦上添花”的枚举，而是组件心智模型的一部分。调用方需要清楚地区分“竞争失败”“本地误用”“远端所有权变化”和“配置错误”。

## 6 关键实现思路

### 6.1 Redis：SET NX + token + watchdog

Redis 方案的主链路是典型的 token 锁模型。加锁时先做一次本地检查，避免同一个 `Locker` 重入同一个 key；然后生成随机 token，用 `SET NX` 抢占锁；成功后把 `key -> token` 写入本地 `locks map`，并启动 watchdog 协程。

解锁不是直接 `DEL`，而是用 Lua 脚本比较 Redis value 和本地 token，只有完全匹配时才删除 key。这个比较是整个 Redis 实现的核心，因为它避免了“锁已经过期并被别人拿走，但旧持有者还把它删掉”的经典误删问题。

watchdog 会按 TTL 的三分之一周期续期。一旦续期失败，或者 Redis 返回 token 不匹配，组件会把这把锁从本地持有状态中移除，并记录“所有权已丢失”。这样做的目的不是让错误消失，而是避免留下“本地以为自己还持有，远端其实早就没了”的僵尸锁状态。

### 6.2 Etcd：Mutex + Session

Etcd 方案不重新设计锁协议，而是直接复用 `clientv3/concurrency` 的 `Mutex` 和 `Session`。默认情况下，一个 `Locker` 持有一个默认 session；如果某次 `WithTTL(...)` 覆盖了 `DefaultTTL`，组件会为这次加锁单独创建 session。

这里一个比较重要的细节是本地状态一致性。Etcd 以前的实现只在真正加锁前检查一次本地状态，这会让同一个 `Locker` 并发抢同一个 key 时出现竞态。现在的实现会在远端加锁成功后再做一次本地检查，如果发现本地已经记录了该 key，就立刻把刚拿到的远端锁释放掉，保证本地“不允许重入”的语义稳定成立。

### 6.3 Close 的真实语义

`Close()` 是这次收紧设计后最关键的地方。之前“关闭 locker”更像是一个半成品概念，现在它被明确成一个生命周期动作：结束当前 `Locker` 的一切持有状态。

对 Redis 来说，这意味着停止每一把锁的 watchdog，并尝试按 token 释放远端 key。对 Etcd 来说，这意味着遍历当前持有的 `Mutex`，执行 `Unlock`，再关闭自定义 TTL session，最后关闭默认 session。`Close()` 仍然不负责底层连接器的关闭，因为连接器不归 `dlock` 所有；但它必须对自己创建和管理的锁状态负责。

## 7 工程取舍与设计权衡

第一个取舍是：`dlock` 追求的是**统一入口**，不是**完全相同的底层协议**。Redis 和 Etcd 的实现天然不同，所以组件不试图把二者伪装成完全等价。例如 TTL 精度就是明确暴露的边界，而不是偷偷帮调用方做向下取整。

第二个取舍是：本地状态不是优化，而是语义的一部分。如果没有 `locks map`，组件仍然能依赖远端完成互斥；但这样调用方会失去“当前 `Locker` 是否已经持有这把锁”的确定性，也无法实现明确的 `ErrLockAlreadyHeld`。因此 `dlock` 选择保留本地状态，同时在 Redis 和 Etcd 两个后端上都收紧本地一致性。

第三个取舍是：`Close()` 必须有实际行为。很多锁组件把关闭做成 no-op，看起来简单，但会把生命周期责任全部甩给调用方。这在基础组件里并不稳。Genesis 更强调资源所有权清晰，因此 `Close()` 至少要对当前 `Locker` 自己创建和维护的状态负责。

最后一个取舍是：不做更复杂的锁抽象。当前没有引入可重入锁、可续租 API、锁状态查询、锁诊断接口等能力。原因不是做不到，而是这些能力一旦加进去，就会显著提高跨后端语义复杂度，也会让 README 和 `go doc` 很难保持清晰。

---

## 8 适用场景与实践建议

`dlock` 适合这些场景：

- 多实例定时任务竞选
- 对某个资源 key 做短时间串行化保护
- 业务侧需要“拿不到锁就跳过”或“等到拿到锁再继续”的简单分支
- 你希望统一 Redis 和 Etcd 的锁接入方式，而不想每个服务重复写 token 校验和 lease 逻辑

它不适合这些场景：

- 需要可重入锁或读写锁
- 需要严格公平锁语义
- 需要长时间人工持有、可观察、可管理的锁平台
- 需要跨语言、跨系统统一制定复杂锁协议

实践上有几个点很重要。第一，始终给 `Lock` 传合理的 `context`，不要让等待无上限。第二，进入关键区后尽快 `defer Unlock(...)`，不要把业务清理逻辑放在解锁前。第三，Etcd 下不要使用子秒级 TTL；这不是“建议”，而是明确的接口约束。第四，把 key 设计成业务可读且有命名空间，例如 `orders:freeze:12345`，避免不同模块发生 key 冲突。

---

## 9 总结

`dlock` 的核心价值，不是提供最复杂的分布式锁能力，而是把最常见、最容易写错的锁语义做成一个小而稳的组件。Redis 方案负责在低门槛场景里提供 token 校验和自动续期，Etcd 方案负责在 lease 模型下提供更自然的一致性语义，而统一的 `Locker` 接口则把调用方最关心的部分收敛成少量方法。

这类组件真正重要的地方，往往不是“能不能加锁”，而是生命周期是否明确、所有权是否可判断、边界失败是否显式。`dlock` 现在的设计重点，就在于把这些地方收紧到更可预测的状态。
