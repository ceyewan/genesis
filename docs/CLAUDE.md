# Genesis 文档撰写指南

本指南用于约束 AI 助手在 Genesis `docs/` 目录下撰写技术博客的行为规范。

## 文档定位

Genesis 文档具有双重目的：

1. **技术推广**：介绍组件的实现细节，推荐开发者使用 Genesis 组件
2. **技术分享**：深入讲解底层原理，面向技术分享和面试场景

## 内容原则

### 语言风格

- **中立**：避免过度营销，客观描述组件能力与边界
- **专业**：使用准确的技术术语，避免口语化表达
- **严谨**：技术描述需准确，不夸大、不模糊
- **简洁**：去除冗余，每句话都有信息量

### 禁止事项

- 禁止使用表情符号（emoji）
- 禁止使用感叹号表达情绪
- 禁止使用"非常"、"极其"、"超级"等程度副词
- 禁止使用自卖自夸的表达（如"完美解决"、"业界领先"）

## 文档结构规范

### 标准章节结构（参考）

```markdown
# 标题：Genesis <组件名>：<简短描述>

## 0. 摘要
- 核心设计的要点罗列（3-6 条）
- 以破折号开头的列表，每条一个独立概念

## 1. 背景：组件要解决的"真实问题"
- 描述微服务场景中的痛点/矛盾
- 说明为什么需要这个组件

## 2. 核心设计：...
- 接口/抽象设计
- 配置模型
- 设计决策

## 3-N. 具体实现细节
- 按子主题分节
- 每节解决一个具体技术点

## N+1. 实践模式与选型建议
- 何时使用
- 何时不使用
- 与其他方案的对比

## N+2. 最佳实践与常见坑
- 推荐用法
- 常见误区

## N+3. 总结（可选）
- 设计取舍总结
- 组件价值定位
```

### 标题规范

- 一级标题：`# Genesis <组件名>：<设计主题>`
- 二级标题：`## <编号>. <主题>`
  - 编号使用阿拉伯数字，如 `## 1. 背景`
- 三级标题：`### <子主题>`
  - 不使用编号

### 代码块规范

```go
// 代码块使用语言标记
func example() {
    // ...
}
```

### 列表规范

- 描述性列表使用破折号 `-`
- 编号列表使用阿拉伯数字 `1.`
- 嵌套列表保持缩进一致

## 内容深度要求

### 必须包含

1. **设计动机**：为什么这样设计
2. **核心抽象**：接口/类型定义
3. **关键流程**：重要操作的执行流程
4. **底层原理**：涉及的标准、协议、算法
5. **边界说明**：能做什么、不能做什么
6. **实践建议**：如何正确使用

### 避免内容

1. 纯 API 文档（应由 README.md 承担）
2. 与 Genesis 无关的通用知识（除非必要背景）
3. 过于细节的代码逐行解释

## 格式规范

### 段落

- 段落间空一行
- 段落首行不缩进
- 段落内不使用强制换行（除非有意为之）

### 表达方式

**优先使用自然段**：对于流程、原理、机制等说明性内容，优先使用自然段落表述，避免过度拆分成列表或代码块。自然段更适合表达逻辑连贯的内容。

**何时使用列表**：仅在以下场景使用列表
- 配置项说明（字段表格）
- 并列的短项（如 3-6 个要点）
- 步骤性操作（编号列表）

**何时使用代码块**：仅在以下场景使用代码块
- 完整的代码示例
- API 接口定义
- 配置文件示例
- 数据结构定义（用于展示类型而非运行逻辑）

避免用代码块展示可以自然语言描述的运行逻辑。

### 强调

- 关键概念使用**粗体**
- 术语首次出现可使用粗体强调
- 代码/命令/文件名使用行内代码块

### 引用

- 引用规范、标准时使用 `[RFC 7519]` 格式
- 引用其他文档使用相对链接

### 分隔线

- 主要章节间使用 `---` 分隔
- 不要过度使用

## 常见术语规范

| 中文 | 英文 | 说明 |
|------|------|------|
| 认证 | Authentication | 验证身份 |
| 鉴权 | Authorization | 验证权限 |
| 中间件 | Middleware | 处理请求/响应的拦截器 |
| 依赖注入 | Dependency Injection | 显式传入依赖 |
| 旁路缓存 | Cache-Aside | 缓存模式 |
| 熔断器 | Circuit Breaker | 容错模式 |
