# Genesis ratelimit：请求入口限流组件的设计与取舍

`ratelimit` 是 Genesis 的治理层（L3）组件，负责在 HTTP、gRPC 和显式业务调用入口前做请求速率控制。它解决的是最常见的一类问题：某个业务键在单位时间内不能无限放行，但又需要允许一定程度的突发流量。本文不重复 README 的快速接入，而是解释 `ratelimit` 为什么保持现在这组接口，为什么分成单机和分布式两种模式，为什么分布式实现要用 Redis 时间，以及为什么错误策略要显式可配。

当前组件的定位很明确：它是**请求入口限流器**，不是完整流量治理平台，也不是多算法限流实验场。它优先保证接口简单、语义清楚和接入成本低。

**摘要**

- `ratelimit` 的核心接口是 `Allow` / `AllowN`，`Wait` 只是单机模式的补充能力。
- 单机模式使用 `x/time/rate`，分布式模式使用 Redis Lua 维护共享桶状态。
- 分布式模式现在仍是令牌桶语义，但会把 `key + limit` 一起编码进 Redis 键，避免不同规则串桶。
- 分布式脚本使用 Redis `TIME` 作为统一时钟，而不是信任业务节点本地时间。
- Gin 和 gRPC 集成默认 `fail_open`，但现在可以显式切到 `fail_closed`。

---

## 1 背景与问题

限流本质上是在“保护系统”与“不过度打扰正常流量”之间找平衡。完全不做限流，热点用户、错误重试和流量尖峰会直接把后端资源顶满；做得太重，又会把一个原本只需要简单入口保护的问题，变成复杂的配额与策略系统。

Genesis 里的 `ratelimit` 选择了一条很克制的路：只解决“给某个键应用一条 `Rate/Burst` 规则”的问题。这个键可以是 IP、用户 ID、RPC 方法名、接口路径，甚至是业务自己拼出来的组合键，但组件本身不试图理解这些维度。调用方提供键和规则，组件负责执行限流判定。

直接使用底层库当然也能完成一部分工作。单机场景下，`golang.org/x/time/rate` 已经足够成熟；分布式场景下，也可以自己写 Redis Lua、ZSET 甚至接入网关限流。但 Genesis 仍然需要一层自己的封装，原因并不在于“底层库不够强”，而在于业务侧真正需要的是一套**统一、稳定、带接入层支持**的治理接口：同样是限流，业务不应该在 Gin、中间件、gRPC 和手写 Redis 脚本之间反复切换心智模型。

---

## 2 基础原理

`ratelimit` 当前的核心算法是**令牌桶（Token Bucket）**。它的直觉很简单：系统按照固定速率往桶里补充令牌，请求需要消耗令牌，没有令牌就拒绝。相比固定窗口，令牌桶允许短时间突发；相比漏桶，它不会把所有请求都强行整形成固定输出速率，更适合接口保护场景。

单机模式直接复用了 `x/time/rate` 的实现，因此这里没有重新造轮子。真正需要做设计判断的是分布式模式。分布式限流最容易踩的两个坑，一个是**跨节点时钟不一致**，另一个是**同一业务键被不同规则复用**。这两个问题在代码层面看起来都不大，但一旦组件对外承诺“这是集群共享限流”，它们就会直接变成语义正确性问题。

---

## 3 设计目标

- **统一入口**：业务只面对 `Limiter` 和中间件/拦截器，不直接依赖具体算法实现。
- **低心智负担**：核心使用方式保持在 `Allow(key, limit)` 这一层，不把复杂配额模型塞给调用方。
- **单机/分布式都可接入**：同样的业务代码可以在不同部署形态下切换实现。
- **边界清晰**：明确 `Wait` 不是分布式能力，错误策略也不是隐式魔法。
- **故障可控**：限流器自身出错时，调用方必须能够明确选择放行还是拒绝。

这些目标决定了 `ratelimit` 不追求“算法大全”，也不追求“把所有治理语义都统一抽象干净”。它要的是一套够用、稳定、容易判断后果的接口。

---

## 4 核心接口与配置

`ratelimit` 的核心接口很小：

```go
type Limiter interface {
	Allow(ctx context.Context, key string, limit Limit) (bool, error)
	AllowN(ctx context.Context, key string, limit Limit, n int) (bool, error)
	Wait(ctx context.Context, key string, limit Limit) error
	Close() error
}
```

这里真正重要的是前两个方法。`Allow` 和 `AllowN` 对两种驱动都成立，语义也最稳定。`Wait` 保留下来，是因为单机场景下它很自然，也确实有用；但 Genesis 不再把它当作“所有驱动都应等价支持”的核心能力。分布式模式下 `Wait` 返回 `ErrNotSupported`，这是组件边界的一部分。

配置上，组件只区分两种驱动：

- `standalone`
- `distributed`

其中 `standalone` 只需要清理间隔和空闲过期时间；`distributed` 当前只需要 Redis key 前缀。这个配置面故意很小，因为规则本身放在每次调用传入的 `Limit` 里，而不是塞到初始化配置里。这样做的好处是：同一个限流器实例可以被不同业务路径复用，但坏处也很明显，分布式模式必须处理“同一个业务键配不同 `Limit`”时的状态隔离问题。

---

## 5 核心概念与错误模型

这个组件最重要的概念其实只有三个：**键**、**规则** 和 **错误策略**。

键决定限流粒度。你可以按 IP、用户、方法名或者任意业务组合键来限流。组件不理解键的业务含义，只把它当状态索引。

规则当前只有两个字段：`Rate` 和 `Burst`。这意味着组件明确站在令牌桶语义上，而不是做成可插拔的多算法平台。这个选择并不是“做不了更多算法”，而是当前没有必要让 API 承担那样的复杂度。

错误策略是这次调整后变得更显式的一点。Gin 中间件和 gRPC 拦截器默认都采用 `fail_open`，也就是限流器自身异常时放行业务请求。这符合很多治理组件“保护业务可用性优先”的默认取舍，但它不应再是隐含行为。因此现在显式引入了 `ErrorPolicy`：

- `fail_open`
- `fail_closed`

这样调用方就能根据场景选择：更偏业务可用性，还是更偏系统保护。

## 6 关键实现思路

### 6.1 单机模式

单机模式的实现非常直接：每个 `key + limit` 对应一个 `x/time/rate.Limiter`，并带一个 `lastSeen` 时间戳，用于后台清理长时间未访问的限流器。这个模式的优点是实现成熟、性能高、没有外部依赖；缺点是只能保护当前进程，无法控制整个集群的总流量。

### 6.2 分布式模式

分布式模式当前仍然使用 Redis Lua 脚本和单键状态来实现共享桶。它不是教科书式“显式存 token 数量”的传统实现，而更接近一种**基于时间债务的令牌桶变体**。脚本里存的是下一次可放行时间，通过时间差反推出是否还有可用令牌。这么做的好处是 Redis 状态更小，脚本逻辑也更紧凑。

这次调整后，分布式实现有两个关键改动。

第一个改动是 Redis key 设计。早期版本只用业务键做 Redis key，这意味着同一个键如果套用了不同的 `Rate/Burst`，会共享同一个桶状态。单机实现并没有这个问题，因为它把 `limit` 也放进了缓存键。现在分布式实现也改成了相同思路：Redis key 会把 `key + rate + burst` 一起编码进去。这样不同规则下的状态天然隔离，单机和分布式的语义终于一致。

第二个改动是统一时钟来源。分布式脚本以前依赖业务节点把本地 `now` 传进 Lua，这对单节点测试完全没问题，但一到多节点部署就会暴露时钟漂移问题。现在脚本改成直接调用 Redis `TIME`，把 Redis 当成共享时钟源。这样虽然不能消除网络延迟带来的全部误差，但至少不会再因为各节点时钟快慢不同而让限流判定失真。

### 6.3 接入层

Gin 和 gRPC 的接入层都故意保持很薄。它们只做三件事：

1. 从请求上下文中提取限流键；
2. 生成本次请求的 `Limit`；
3. 根据错误策略决定放行、拒绝还是降级。

这样设计的好处是中间件和拦截器没有把业务策略写死；真正的限流粒度和规则仍由调用方决定。代价是默认行为必须非常清晰，否则“无 key 放行”“无效规则放行”“限流器出错放行”这些降级路径就会变成难以察觉的隐式逻辑。也正因为如此，这次把错误策略显式化是必要的。

## 7 工程取舍与设计权衡

### 7.1 为什么不做多算法可切换

从实现角度看，Redis 上继续做滑动窗口、漏桶并不难；真正困难的是把这些算法统一进一套清晰、稳定、低心智负担的 API。不同算法对配置字段、返回信息、剩余量计算和排障方式的要求都不同。如果现在为了“功能更全”把它们都塞进 `ratelimit`，很容易把一个本来简单的入口保护组件，变成语义含混的算法平台。

Genesis 当前更看重的是明确边界：`ratelimit` 就是令牌桶语义的入口限流器。以后如果要扩展多算法，也应该在需求足够稳定之后再做，而不是先把配置面做大。

### 7.2 为什么保留 `Wait`

从抽象纯度看，最干净的做法也许是直接删掉 `Wait`，因为分布式模式不支持它。但从工程现实看，单机模式下 `Wait` 仍然是合理且常用的能力，删掉反而会让单机接入变差。因此 Genesis 现在的选择不是强行做统一，而是明确承认“`Wait` 只属于单机场景”。

### 7.3 为什么默认 `fail_open`

默认 `fail_open` 的背后是一个偏可用性的判断：限流器本身属于治理层，如果因为 Redis 故障或者限流脚本异常直接把业务全拒掉，很多场景下损失会比暂时放开限流更大。但这不是普适真理。有些系统更重视保护而不是业务连续性，比如某些防刷、风控或昂贵下游资源保护场景，这时 `fail_closed` 才是正确策略。

因此最合理的设计不是强推其中一个，而是：**给出稳定默认值，同时把选择权显式交给调用方。**

## 8 适用场景与实践建议

`ratelimit` 适合的典型场景包括：

- HTTP API 入口按 IP 或用户限流；
- gRPC 方法级入口保护；
- 网关之后、业务逻辑之前的轻量防刷；
- 单实例服务的本地资源保护；
- 多实例服务共享 Redis 的全局入口控制。

它不适合的场景包括：

- 需要复杂多维配额和额度结转；
- 需要精确剩余令牌、重试时间等强观测接口；
- 需要分布式排队、阻塞获取令牌；
- 需要多种限流算法按配置自由切换。

实践上，最值得注意的几点是：

第一，键设计比算法更重要。限流器只能按你给的键工作。错误的键粒度会导致“全局误伤”或“保护失效”。

第二，分布式模式下不要假设所有规则天然共享状态。当前实现已经把 `limit` 纳入 Redis key，这意味着同一个业务键在不同规则下是隔离的。这个行为通常是正确的，但如果你的业务需要“同一键在多处共用同一配额”，就应该自己统一规则，而不是依赖组件帮你合并。

第三，默认 `fail_open` 并不代表它总是最佳实践。凡是限流承担了硬保护职责的地方，都应该明确评估是否要切换到 `fail_closed`。

---

## 9 总结

Genesis 的 `ratelimit` 没有试图成为一个复杂的流量治理系统，它更像一块明确、克制、可组合的积木：给你一个稳定的入口限流接口，再配上 Gin 和 gRPC 的接入层。

它当前最重要的设计判断有三个：坚持小接口、坚持令牌桶语义、坚持把关键治理行为显式化。分布式模式使用 Redis 时间、按 `key + limit` 隔离桶状态，以及错误策略可配置，都是围绕这三个判断做出的收敛。对 Genesis 来说，这比盲目扩张算法种类更重要。
